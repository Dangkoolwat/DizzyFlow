# 🚀 DizzyFlow 2.1: Final Master Plan
"외부 전사 결과를 정리하고, 후속 보정으로 자막의 완성도를 높인다."

## 1. 개요 및 핵심 비전

DizzyFlow 2.1은 특정 전사 엔진에 종속되지 않는다.  
Whisper 계열을 포함한 외부 전사 프로그램이 생성한 timed transcript를 받아,  
후속 보정(Post-Correction) 패스를 통해 더 읽기 쉽고 더 정리된 자막 결과로 다듬는 단계다.

2.1의 핵심은 다음과 같다.

- 전사 엔진 자체보다 결과 정제 워크플로우에 집중한다
- 저사양 기기에서도 안정적으로 동작하도록 순차 처리 구조를 유지한다
- 자동 보정 결과를 원본과 비교 가능한 형태로 제공한다
- 만족 시 즉시 preview / export로 이어질 수 있다

즉 DizzyFlow 2.1은 AI 편집기가 아니라,  
외부 전사 결과를 안전하게 정리하고 비교·검토할 수 있게 하는 후속 보정 단계다.

---

## 2. 처리 철학

2.1에서는 실시간 병렬 보정을 기본값으로 두지 않는다.  
외부 전사 프로그램의 처리만으로도 시간이 오래 걸릴 수 있으므로,  
후속 보정은 전사 완료 이후 별도 패스로 실행하는 구조를 채택한다.

이 방식의 목적은 다음과 같다.

- 무거운 병렬 처리 리스크를 줄인다
- 완료 시간을 더 예측 가능하게 만든다
- 저사양 장치에서도 workflow 안정성을 유지한다
- 사용자가 원본 결과와 보정 결과를 명확히 비교할 수 있게 한다

즉, 2.1의 기본 흐름은 다음과 같다.

1. 외부 전사 프로그램이 초안 자막을 생성한다
2. 필요 시 후속 보정 패스가 실행된다
3. 결과 화면에서 원본과 보정본을 함께 확인한다
4. 사용자는 바로 preview / export 하거나, 이후 refinement 단계로 넘어갈 수 있다

---

## 3. 오디오/싱크 기준

DizzyFlow는 후속 정렬과 물리 싱크 보조를 위해 다음 기준을 내부적으로 유지할 수 있다.

- 데이터 규격: 16kHz, 16-bit, Mono WAV
- 고정 데이터 레이트: 1초 = 32,000 Bytes
- 기준 공식: `offset = time * 32,000`

이 기준은 제품의 전면 기능이라기보다,  
향후 정밀한 싱크 처리와 오디오 재참조 기능을 위한 내부 정렬 기준이다.

즉 2.1 단계에서는 이것을 사용자 기능으로 과장하지 않고,  
문서와 처리 파이프라인의 기술적 기반으로 유지한다.

---

## 4. 2.1 자동 보정 규칙

2.1의 후속 보정은 과도한 자유 편집이 아니라,  
미리 정의된 정제 규칙을 적용하는 수준으로 제한한다.

### 4.1 Clean Text
외부 전사 결과에 포함될 수 있는 불필요한 기호나 노이즈를 정리한다.

예:
- 마침표(`.`) 제거
- 음악 기호 제거 (`♪`, `♫`)
- 배경음/설명성 괄호 표현 제거 (`[...]`, `(...)`)
- 의미 없는 구분선 제거 (`---`)
- 과도한 느낌표/물음표 정리 (`!!!`, `???`)
- 불필요한 따옴표 및 반복 문장부호 정리

목표는 장식적 정리가 아니라,  
자막 결과의 시각적 무결성과 안정성 확보다.

### 4.2 Semantic Split
텍스트를 말의 호흡과 의미 단위에 맞춰 정리한다.

예:
- 자연스러운 줄바꿈 삽입
- 최대 2줄 자막 원칙 고려
- 지나치게 긴 한 줄 자막 정리

목표는 단순 개행이 아니라,  
읽기 쉬운 자막 구조 생성이다.

### 4.3 Optional Structured Output
필요한 경우 후속 보정 패스는 원문 외에 추가 결과층을 생성할 수 있다.

예:
- 정리된 메인 텍스트
- 선택적 번역 프리셋 결과
- 규칙 적용 정보

다만 2.1 단계에서는 번역/이중언어 출력이 기본 규칙은 아니다.  
이는 선택적 preset 또는 이후 단계 확장으로 다룬다.

## 5. Gemma 설정 원칙

Gemma 기반 후속 보정은 Settings에서 on/off 가능한 선택 기능으로 둔다.

### OFF
- 외부 전사 결과만 사용
- 비교 화면 없이 기본 결과 검토
- 바로 preview / export 가능

### ON
- 외부 전사 완료 후 후속 보정 패스 실행
- 미리 설정된 규칙/프롬프트 적용
- 결과 화면에서 원본과 보정본 비교 가능

여기서 중요한 것은 사용자가 모델을 직접 다루는 것이 아니라,  
보정 패스를 사용할지 여부를 고른다는 점이다.

즉 사용자 경험의 중심은 모델 선택이 아니라  
결과 정제 workflow 선택이어야 한다.

## 6. 동작 프로세스

### 6.1 Gemma OFF
1. 외부 전사 프로그램 실행
2. timed transcript 생성
3. 결과 화면 표시
4. preview / export 가능

### 6.2 Gemma ON
1. 외부 전사 프로그램 실행
2. timed transcript 생성
3. 후속 보정 패스 실행
4. 원본 / 보정 결과 비교 화면 표시
5. preview / export 가능

핵심은 전사 결과가 먼저 기준 데이터로 존재해야 한다는 점이다.  
후속 보정은 그 위에 얹히는 추가 패스이며,  
기본 전사 결과를 덮어쓰는 구조가 아니다.

## 7. 결과 구조

2.1에서는 하나의 작업 루트 아래에 원본 결과와 후속 결과를 함께 유지할 수 있다.

예:

    작업명
        초안 · 2026-04-06 14:20
        자동 보정 · 2026-04-06 14:24

또는 결과 화면에서 다음처럼 표시할 수 있다.

- Original
- Corrected

이 구조의 목적은 다음과 같다.

- 원본 보존
- 보정 결과 비교
- 만족 시 즉시 선택/내보내기
- 이후 refinement 확장 기반 마련

---

## 8. 내부 데이터 예시

    {
        "segment_id": "seg_2026_04_06_001",
        "sync": {
            "start": 320000,
            "end": 512000
        },
        "original": "♪ (음악 소리) ...안녕하세요. 오늘 날씨가 참 좋네요!!!",
        "refined": {
            "main": "안녕하세요 오늘 날씨가 참 좋네요"
        },
        "metadata": {
            "noise_removed": ["♪", "(음악 소리)", "...", "!!!"],
            "period_removed": true,
            "rules_applied": ["clean_text", "semantic_split"]
        }
    }

이 예시는 설명용이다.  
핵심은 다음을 만족하는 것이다.

- 원본 전사 결과 보존
- 보정 결과 별도 보관
- 적용 규칙 추적 가능
- 이후 refinement 확장 가능

## 9. 사용자 액션

2.1 단계에서 사용자는 다음을 할 수 있다.

- 원본 결과 확인
- 보정 결과 확인
- 비교 화면 검토
- 결과 preview
- 결과 export / download

즉 2.1은 자동 보정 결과를 보고 바로 활용할 수 있는 단계다.

더 세밀한 수동 수정이나 부분 재처리는 2.2의 영역으로 넘긴다.

---

## 10. 범위 제한

2.1에서는 다음을 하지 않는다.

- 실시간 병렬 보정 필수화
- 자유 대화형 편집
- 선택 구간 반복 재보정
- 오디오 재참조 딥 픽스
- 복잡한 버전 트리
- 단어 단위 부분 승인/병합

즉 2.1은 어디까지나  
자동 후속 보정 + 비교 + 즉시 활용 단계에 머문다.

선택적으로 활성화된 후속 보정 패스를 적용해  
더 정리된 자막 결과를 원본과 비교 가능한 형태로 제공하는 단계다.

# 문제 분석

## 1. 개요
DizzyFlow 초기 프로토타입은 단순한 4단계 상태 머신과 중첩된 사이드바 구조를 가지고 있었으나, 이는 프로젝트의 "Workflow-first" 철학과 최신 UX 가이드라인(`docs/ux/`)을 충분히 반영하지 못하고 있었습니다.

## 2. 상세 문제점
- **상태 머신 미비**: `idle`, `ready`, `processing`, `completed` 4단계만 존재하여 `failed`, `cancelled`와 같은 터미널 상태 처리가 부족함.
- **사이드바 뎁스**: `Documents` 섹션 내부에 리스트가 숨겨져 있어 사용자가 여러 작업의 상태를 한눈에 파악하기 어려움.
- **Safe Lock 부재**: 작업 처리 중 사이드바 이동이나 설정 변경이 차단되지 않아 데이터 정합성 및 안정성 문제 발생 가능.
- **시각적 정보 부족**: 파일 드롭 후 "Ready" 상태에서 파일명 외에 재생 시간, 용량, 포맷 등 사용자에게 확신을 줄 수 있는 메타정보가 표시되지 않음.
- **빌드 호환성**: macOS 15+ 전용 API(`onScrollPhaseChange`) 사용으로 인해 타겟 버전(14.6)에서 빌드 실패 발생.

## 3. 제약 사항
- 실제 엔진(Sherpa-onnx 등) 통합 전이므로 모든 데이터는 Mock으로 처리해야 함.
- 파일 I/O 시스템 구축 전이므로 파일명 기반의 Mock 메타데이터 생성 로직 필요.

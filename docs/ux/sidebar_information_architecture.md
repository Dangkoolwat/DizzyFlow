## 작업 루트 이름 표시 원칙

상위 루트는 원본 파일을 기준으로 생성되지만,  
UI에 표시되는 이름은 전체 파일명이 아니라 축약된 작업명이어야 한다.

즉:

- 내부 식별자: 원본 파일 전체 경로/전체 파일명 유지
- 사용자 표시명: 짧고 식별 가능한 작업명 사용

---

## 왜 파일명을 그대로 쓰면 안 되는가

영상 원본 파일명은 다음 문제가 있다.

- 이름이 너무 길 수 있음
- 날짜, 기기명, 해상도, 버전 문자열이 과도하게 섞임
- 사용자가 한눈에 의미를 파악하기 어려움
- 사이드바에서 폭을 지나치게 차지함

따라서 UI에서는 원본 파일명 전체를 그대로 노출하지 않는다.

---

## 권장 방식

표시명은 파일명에서 자동 추출하되,  
다음 기준으로 최소 의미 단위만 남기는 것이 좋다.

우선순위는 다음과 같다.

1. 사용자가 바로 알아볼 핵심 단어
2. 작업을 구분할 수 있는 최소 식별어
3. 너무 긴 기술적 꼬리표 제거

예:

    Interview_with_Kim_2026_final_audio_mix_v03.mov
    -> Interview with Kim

    YT_Live_2026_04_06_EP12_master_export_4k.mov
    -> YT Live EP12

    clientA_promo_cut_ver2_final_really_final_001.mp4
    -> clientA promo cut

---

## 내부/외부 분리

### 내부적으로 유지할 것
- full original filename
- file path
- import metadata

### 사용자에게 보여줄 것
- short display title
- 필요 시 subtitle 또는 secondary metadata

즉 파일명 전체는 저장하되,  
사이드바와 주요 UI에는 짧은 display title만 쓴다.

---

## 추천 UI 구조

    Interview with Kim
        작업 1 · 2026-04-06 14:20
        작업 2 · 2026-04-06 15:05

또는

    YT Live EP12
        초안 · 2026-04-06 14:20
        자동 보정 · 2026-04-06 15:05

이렇게 하면 상위 루트는 짧고, 하위 결과 항목은 시간/성격으로 구분된다.

---

## 표시명 생성 규칙 제안

자동 생성 시 다음 규칙을 고려할 수 있다.

- 확장자 제거
- underscore, hyphen을 공백으로 정리
- `final`, `master`, `export`, `v03`, 해상도 표기 같은 기술 꼬리표 제거
- 너무 길면 앞쪽 의미 단위만 유지
- 필요 시 숫자 에피소드/회차는 유지
- 최종 표시 길이는 짧게 제한

즉 목표는 “원본 파일명을 충실히 재현”하는 것이 아니라,  
사용자가 이 작업을 바로 알아볼 수 있는 짧은 제목을 만드는 것이다.

---

## 핵심 원칙

상위 루트 이름은 파일명 기반으로 생성되더라도,  
UI에서는 의미 중심의 축약 작업명으로 보여줘야 한다.

한 줄 요약:

원본 파일명은 내부 식별용으로 보존하고, 사용자에게는 최소 의미 단위로 정리된 짧은 작업명을 표시한다.

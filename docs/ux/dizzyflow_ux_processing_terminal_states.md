# DizzyFlow UX Flow Notes

## Processing and Terminal States
Last updated: 2026-04-04

이 문서는 DizzyFlow 초기 프로토타입의 Processing 단계와
작업 종료 상태(Completed / Failed / Cancelled)에 대한 UX 기준을 정리한 문서다.

다른 에이전트가 구현 시 오해하지 않도록,
헤비한 작업을 안전하게 보호하는 UX 원칙과
종료 상태별 Workspace 동작을 함께 설명한다.

---

## 1. 핵심 전제

DizzyFlow의 전사 작업은 CPU / GPU 사용량이 큰 헤비한 작업이다.

따라서 Processing 단계에서는
사용자가 다른 문서를 클릭하거나 설정을 바꾸는 등의 동작으로 인해
앱 상태가 꼬이지 않도록 보호하는 설계가 필요하다.

초기 프로토타입 기준 Processing 단계의 핵심 원칙은 다음과 같다.

    작업이 시작되면 앱은 Safe Lock 상태로 들어간다
    Processing 중에는 대부분의 UI 인터랙션을 차단한다
    취소(Job Cancel)만 유일하게 열어둔다
    사용자는 작업이 멈춘 것이 아니라 진행 중임을 분명하게 인지할 수 있어야 한다

---

## 2. Processing 단계 정의

Processing은 사용자가 Ready 상태에서
"시작하기" 버튼을 눌러 실제 작업이 시작된 상태다.

이 시점에 다음이 함께 발생한다.

    새 Document 폴더 생성
    사이드바에 해당 문서 항목 추가
    실제 엔진 작업 시작
    Workspace가 Processing UI로 전환

Processing은 단순 로딩 화면이 아니라,
"작업이 안전하게 실행되는 동안 사용자의 오동작을 막고,
동시에 진행 상황을 안심감 있게 보여주는 단계"다.

---

## 3. Safe Lock 전략

Processing 중에는 Safe Lock 전략을 적용한다.

이 전략의 목적은 두 가지다.

    1. 작업 중 앱 상태가 꼬이지 않게 보호
    2. 앱이 멈춘 것이 아니라 열심히 처리 중이라는 신뢰 제공

### 잠그는 대상

다음 UI는 Processing 중 비활성화한다.

    사이드바
    상단 영역의 설정 변경 기능
    인스펙터 내 설정 변경 기능
    그 외 작업 흐름을 깨뜨릴 수 있는 입력 요소

비활성화 방식은 다음 기준을 따른다.

    반투명 또는 Dim 처리
    클릭 불가
    현재 사용할 수 없는 상태임이 시각적으로 명확해야 함

### 열어두는 대상

Processing 중 유일하게 적극적으로 열어두는 액션은 다음이다.

    Job Cancel 버튼

즉 Processing은
"취소만 가능한 보호된 상태"로 본다.

---

## 4. 상단 영역 역할 변화

상단 영역은 단계에 따라 역할이 달라진다.

### Idle / Ready

Idle과 Ready에서는 상단 영역이 설정 UI 역할을 한다.

표시/수정 가능한 항목:

    FPS
    언어
    FCP 템플릿
    전사 모델

### Processing 이후

Processing부터는 상단 영역이 더 이상 설정 UI가 아니다.

이 시점부터 상단 영역은
주요 작업 메시지를 보여주는 표시 영역으로 전환된다.

예시:

    오디오 분석 중...
    VAD 분석 중...
    전사 생성 중...
    SRT 생성 중...
    FCPXML 생성 중...
    작업 완료
    작업 실패
    사용자 취소

즉 설정은 Idle / Ready까지만 직접 조정하고,
Processing 이후에는 메시지 중심 영역으로 사용한다.

---

## 5. Processing 단계의 중앙 Workspace 구성

Processing 단계에서 중앙 Workspace는
단순 바람개비를 보여주는 공간이 아니다.

사용자는 이 화면을 통해
작업이 실제로 진행되고 있다는 확신을 받아야 한다.

중앙 Workspace에는 다음 요소를 둔다.

    전용 진행 바
    현재 단계 메시지
    누적 스크롤형 실시간 결과 영역
    Job Cancel 버튼

### 5.1 전용 진행 바

시스템 바람개비(Beachball) 대신
DizzyFlow 전용 진행 바를 사용한다.

이유:
시스템 로딩 인디케이터는 사용자가 앱이 멈췄다고 오해할 수 있다.

따라서 Processing 화면은 다음처럼 구성한다.

    화면 전체를 살짝 어둡게(Dim)
    중앙의 진행 영역은 밝고 선명하게 유지
    사용자가 "작업이 살아 있다"고 느낄 수 있어야 함

### 5.2 현재 단계 메시지

현재 어떤 처리가 진행 중인지 텍스트로 보여준다.

예시:

    오디오 분석 중...
    VAD 분석 중...
    전사 생성 중...
    SRT 생성 중...
    FCPXML 생성 중...

이 메시지는 진행률과 함께 사용자의 불안을 줄이는 핵심 요소다.

### 5.3 누적 스크롤형 실시간 결과 영역

엔진이 문장을 생성할 때마다
실시간 자막 내용이 중앙 영역에 계속 누적되어 보여야 한다.

이 영역은 스크롤이 필수다.

이유:

    10분이 넘는 긴 데이터도 입력될 수 있음
    분할 처리된 개별 파일 흐름도 사용자가 확인할 수 있어야 함
    전체가 실제로 생성되고 있다는 신뢰를 주어야 함

즉 최신 몇 개만 보여주는 구조가 아니라,
전체 흐름을 누적 스크롤로 볼 수 있는 구조를 기본으로 한다.

### 5.4 자동 스크롤 규칙

실시간 결과 영역은 기본적으로
가장 최근에 생성된 문장으로 자동 스크롤된다.

이 규칙의 목적은 다음과 같다.

    사용자가 앱이 멈추지 않고 계속 일하고 있음을 즉시 체감
    최신 생성 결과를 자연스럽게 따라갈 수 있음

단, 사용자가 직접 위로 스크롤하여 과거 내용을 보고 있는 경우에는
자동 스크롤을 강제로 유지하지 않는다.

권장 규칙:

    기본 상태에서는 최신 결과로 자동 스크롤
    사용자가 수동 스크롤을 시작하면 자동 스크롤 일시 중지
    다시 맨 아래 근처로 돌아오면 자동 스크롤 재개

즉 자동 스크롤은 "도움"이어야 하며,
사용자의 읽기 흐름을 방해하면 안 된다.

### 5.5 현재 처리 위치 하이라이트

현재 엔진이 처리 중인 segment 또는 자막 카드는
시각적으로 하이라이트할 수 있다.

예시 방식:

    현재 카드의 배경 강조
    현재 카드의 테두리 강조
    현재 위치를 보여주는 안정적인 색 변화

이 하이라이트의 목적은 다음과 같다.

    전체 결과 중 지금 어디를 처리하고 있는지 직관적으로 보여줌
    누적 스크롤 구조 안에서도 현재 진행 위치를 쉽게 파악하게 도움
    "앱이 실제로 일하고 있다"는 신뢰를 강화

초기 버전에서는 과도한 애니메이션보다
안정적이고 분명한 시각 표시를 우선한다.

### 5.6 Job Cancel 버튼

Processing 중 유일하게 활성화된 버튼이다.

배치 원칙:

    눈에 잘 띄어야 함
    사용자가 언제든 탈출할 수 있다는 심리적 안정감을 줘야 함
    다른 요소에 묻히지 않도록 선명한 시각적 우선순위를 가짐

정확한 위치는 구현 단계에서 조정 가능하지만,
중앙 작업 영역 안에서 쉽게 찾을 수 있는 위치에 둔다.

## 6. Processing 중 Sidebar 동작

시작하기 버튼을 누르면 새 Document 항목이 사이드바에 추가된다.

Processing 중 사이드바는 클릭할 수 없지만,
현재 작업 문서가 존재한다는 것은 시각적으로 확인 가능해야 한다.

권장 방식:

    현재 문서 항목 표시
    processing 상태를 알 수 있는 배지 또는 로딩 표시 추가

이유:
사용자가 현재 어떤 문서가 작업 중인지 직관적으로 이해할 수 있어야 하기 때문이다.

---

## 7. Processing 중 Inspector 역할

Processing 중 Inspector는 설정 변경 공간이 아니라,
작업의 상세 맥락을 보여주는 읽기 전용 패널이다.

표시할 수 있는 정보 예시는 다음과 같다.

    현재 단계
    전체 진행률
    입력 파일 정보
    현재 적용 중인 설정 정보
    단계별 진행 상태

ETA(남은 예상 시간)는 있으면 좋지만,
초기 버전에서는 정확도 문제가 있을 수 있으므로 필수로 두지 않는다.

즉 1차 버전 기준으로는 다음 정도면 충분하다.

    단계 리스트
    전체 진행률
    파일 / 설정 정보

## 8. Cancel 직후 UX

사용자가 Job Cancel 버튼을 누르면
취소 의사가 즉시 반영되어야 한다.

### 8.1 즉각 반응

취소 버튼을 누르는 순간
다음과 같은 짧은 메시지를 사용자에게 보여줄 수 있다.

    작업을 취소하는 중...

이 메시지는 중간 과정을 길게 만드는 목적이 아니라,
사용자 입력이 반영되었음을 알려주는 짧은 피드백이다.

### 8.2 취소 처리 후 화면 전환

엔진 정리가 끝나면
Processing 상태를 종료하고
해당 문서의 Cancelled 화면으로 전환한다.

중앙 Workspace에는 다음을 보여준다.

    사용자 취소 안내 메시지
    Restart 버튼

예시 문구:

    사용자가 취소한 작업입니다.

### 8.3 Sidebar 상태 유지

취소된 문서는 사이드바에 남아 있어야 한다.

권장 방식:

    Cancelled 배지 표시
    사용자가 나중에 다시 선택 가능
    Restart를 통해 Processing을 다시 수행할 수 있음

초기 버전 기준 Restart는 resume가 아니라 rerun으로 본다.

즉 다음과 같다.

    Cancelled
    -> Restart
    -> Processing
    -> Completed 또는 Failed

## 9. Completed 종료

작업이 정상 완료되면
Processing의 Safe Lock 상태는 사라지고,
Completed 화면이 나타난다.

이 전환은 자동으로 일어난다.

즉 사용자는 별도 조작 없이
바로 결과 화면으로 진입한다.

### 9.1 Completed 화면의 핵심 목적

Completed 화면은
작업 완료를 명확히 알려주고,
결과물을 바로 사용할 수 있게 하는 것이 핵심이다.

### 9.2 Completed 화면의 주요 액션 우선순위

초기 버전 기준 액션 우선순위는 다음과 같다.

    1. FCPXML 다운로드
    2. SRT 다운로드
    3. 작업 미리보기
    4. 새 작업

이 우선순위의 이유는 다음과 같다.

    DizzyFlow의 핵심 목적은 Final Cut Pro workflow 지원
    따라서 FCPXML이 가장 중요한 결과물
    SRT는 표준 자막 파일로서 매우 중요
    미리보기는 검수용 유틸리티
    새 작업은 다음 흐름으로 빠르게 넘어가기 위한 액션

### 9.3 Completed 화면의 표시 내용

중앙 Workspace에는 다음을 보여줄 수 있다.

    작업 완료 메시지
    최종 SRT 전체 미리보기
    다운로드 버튼
    새 작업 버튼

최종 SRT는 전체가 하나의 파일로 합쳐진 결과를 기준으로 보여준다.

이 영역은 스크롤이 필수다.

이유:

    10분 이상 데이터도 최종적으로 하나의 SRT로 합쳐짐
    사용자는 전체 결과를 확인용으로 훑어볼 수 있어야 함

Inspector는 Completed 상태에 맞게 다음 정보로 갱신된다.

    완료 상태
    결과물 정보
    생성 관련 메타정보

### 9.4 검수 지원: 저신뢰 구간 시각 가이드

이 항목은 DizzyFlow의
"Review는 Optional이다" 철학을 실현하기 위한 확장 포인트다.

목적:

    전체 자막을 모두 검수하도록 강요하지 않음
    AI가 헷갈렸을 가능성이 높은 구간만 가볍게 표시
    사용자가 필요한 부분만 빠르게 확인 가능

가능한 표현 방식 예시:

    노란색 밑줄
    작은 경고 아이콘(예: ⚠️)
    특정 문장에 대한 시각적 강조 표시

이 기능은 Completed 단계의 결과 미리보기에서 우선 고려한다.

초기 버전에서는 문서 수준의 방향성으로 유지하고,
구체 구현은 confidence score의 출처와 기준이 정리된 뒤에 진행한다.

## 10. Failed 종료

작업이 Processing 중 오류로 종료되면
해당 문서는 Failed 상태로 저장된다.

사이드바에서 이 문서를 선택하면
중앙 Workspace는 Failed 화면을 보여준다.

### 10.1 Failed 화면의 목적

초기 버전 기준 Failed 화면의 목적은 다음과 같다.

    사용자가 실패를 인지할 수 있음
    어느 단계에서 문제가 발생했는지 확인 가능
    디버깅성 정보 확인 가능

### 10.2 Failed 화면의 정보 우선순위

Failed 상태의 핵심 정보는
Inspector보다 먼저 중앙 Workspace에서 명확히 보여주는 것을 우선한다.

이유:

    사용자가 실패 사실과 실패 지점을 즉시 이해해야 함
    실패 정보가 보조 패널에만 있으면 너무 숨겨져 보일 수 있음
    "내 파일 문제인지, 앱 문제인지"를 빠르게 판단하게 도와야 함

즉 중앙 Workspace에는 최소 다음을 우선적으로 보여준다.

    실패 상태 제목
    실패 단계
    짧은 설명
    새 작업 버튼

Inspector는 보조 정보 또는 추가 디버깅 정보를 보여주는 역할로 둔다.

### 10.3 표시 예시

    오류 메시지
    실패 단계
    필요 시 추가 디버깅 정보

초기 버전에서는
개발자용 raw log 전체를 그대로 노출하기보다,
사용자가 이해 가능한 수준의 실패 지점 안내를 우선한다.

예시:

    오디오 분석 실패
    VAD 처리 실패
    전사 엔진 실행 실패
    SRT 생성 실패
    FCPXML 생성 실패

### 10.4 Failed 화면의 기본 액션

Failed 화면의 기본 액션은 다음과 같다.

    새 작업

초기 버전에서는 Failed 상태에 Restart를 기본 제공하지 않는다.

이유:

    내부 버그나 파이프라인 문제일 경우 무의미한 반복 재시도를 유도할 수 있음
    사용자가 같은 조건으로 계속 시도하며 더 큰 피로를 느낄 수 있음
    cancel과 fail은 성격이 다름

즉 초기 버전 기준으로는 다음처럼 구분한다.

    Cancelled
    -> Restart 제공

    Failed
    -> Restart 기본 비제공
    -> 새 작업 제공

## 11. Terminal States 정리

초기 프로토타입에서는
성공 / 실패 / 취소를 별도 상태 모델로 더 복잡하게 세분화하기보다,
WorkflowPhase의 종료 지점(Terminal States)으로 본다.

즉 현재 기준은 다음과 같다.

    Idle
    Ready
    Processing
    Completed
    Failed
    Cancelled

코드 수준에서도 `WorkflowPhase`를 위 6단계 구조로 동기화하는 방향을 권장한다.

예시:

    idle
    ready
    processing
    completed
    failed
    cancelled

이 구조는 각 종료 상태에 따라
Workspace UI를 다르게 보여주기 쉽다는 장점이 있다.

## 12. 단계별 요약

### Processing

    Safe Lock 적용
    대부분의 인터랙션 차단
    Job Cancel만 활성화
    전용 진행 바 + 현재 단계 메시지 + 누적 스크롤형 실시간 결과 표시
    기본적으로 최신 결과로 자동 스크롤
    사용자가 수동 스크롤 중이면 자동 스크롤 일시 중지
    현재 처리 중인 segment는 시각적으로 하이라이트 가능
    상단 영역은 설정 UI가 아니라 메시지 영역
    사이드바에는 현재 작업 문서가 표시되지만 클릭 불가

### Completed

    작업 성공 종료
    최종 SRT 전체 스크롤 미리보기
    FCPXML 다운로드 우선
    SRT 다운로드
    작업 미리보기
    새 작업
    저신뢰 구간 가이드는 향후 확장 포인트

### Failed

    오류 종료
    중앙 Workspace에 실패 상태와 실패 지점을 우선 표시
    기본 액션은 새 작업
    Restart 기본 비제공

### Cancelled

    사용자 취소 종료
    취소 안내 메시지와 Restart 버튼 표시
    다시 시작 시 rerun으로 Processing 재진입
